# Vibe Coding 工业级开发标准

## 核心理念

Vibe Coding = 用自然语言指挥 AI 写代码，但必须有规矩。

AI 像一个极其聪明但极度粗心的实习生。不给规矩，它会：
- 一口气写 20 个文件，你不知道哪个错了
- 编造不存在的接口和函数
- 你刚改好的地方，它又悄悄改坏了

本标准将你从"AI 的傀儡"变成"AI 的指挥家"。

✅ 适合：正式上线产品、长期迭代应用、遗留系统重构、团队协作项目
❌ 不适合：Hackathon、一次性脚本、5 分钟 Demo

## 8 阶段开发流程（必须串行，禁止跳步）

每阶段结束需要**人工确认**才能进入下一阶段。

### 阶段1：需求收敛
1. 用 <=100 字描述项目用途、目标用户、核心痛点
2. 划分核心/次要/扩展功能（核心最多 3 个）
3. 判定项目体量（小型/中型/大型）与运行载体（Web/移动端/桌面/CLI）

**门禁**：人工确认需求文档

### 阶段2：架构锁档（⚠️ 一旦确认，后续不得私自修改）
1. 绘制 Mermaid 业务流程图
2. 模块拆分、定义接口
3. 锁定目录结构
4. 整理依赖清单

**门禁**：人工确认并锁定 → `git commit -m "[Stage-2] 锁档"`

### 阶段3：环境对齐
1. 验证 CLAUDE.md 已生效
2. 确认增量开发模式已理解

**门禁**：人工确认

### 阶段4：脚手架初始化（⚠️ 硬性门槛）
1. 初始化项目、安装依赖、清理冗余
2. **空项目必须成功启动且无报错，否则绝对禁止进入编码阶段**

**门禁**：空项目启动成功 → `git commit -m "[Stage-4] 脚手架"`

### 阶段5：单模块增量编码
- 每次只开发一个最小功能模块
- 生成 → 运行验证 → 确认通过 → 提交 → 下一个模块
- **禁止**：批量生成、多模块并行

**门禁**：每模块验证通过 → `git commit -m "[Stage-5] 模块：{名称}"`

### 阶段6：单模块全场景测试
测试清单：正常流程、空数据、非法输入、超时、断连、Loading、Error、Empty

**门禁**：覆盖率 ≥ 80%

### 阶段7：全局联调 + 重构
全链路联调、统一风格、清理死代码、性能优化

**门禁**：lint 通过 + 全量测试通过

### 阶段8：打包交付
打包、生成 README、制定迭代规划

**门禁**：人工确认交付物完整

## 7 条铁律（强制执行）

| # | 铁律 | 对抗问题 | 拦截动作 |
|---|------|----------|----------|
| 1 | 禁止批量生成多模块 | AI 惰性 | 阻断+回滚，请单模块开发 |
| 2 | 阶段2定稿不得私自修改 | 架构随意变动 | 阻断，需回阶段2重新定稿 |
| 3 | 严禁编造不存在的接口 | AI 幻觉 | 阻断+拒绝，要求先确认接口存在 |
| 4 | 必须补充完整状态处理 | 只写快乐路径 | 阻断，请补充 loading/empty/error |
| 5 | 已验收模块禁止随意修改 | 修复引入新 Bug | 阻断+确认，需走变更流程 |
| 6 | 空项目初始化失败禁止编写业务代码 | 在错误基础上建造 | 绝对阻断 |
| 7 | 禁止编写死代码、占位代码 | 代码腐烂 | 警告+清理 |

## 常见陷阱

| 陷阱 | 解决方案 |
|------|----------|
| AI 跳过阶段直接生成代码 | 指出"你跳过了阶段X"，要求回到指定阶段 |
| AI 批量生成多个文件 | 立即停止，`git reset --hard HEAD~1`，只保留单模块 |
| AI 编造不存在的接口 | 要求 AI 先输出接口清单，人工核实后再编码 |
| 阶段2锁档后需要修改 | 回到阶段2重新定稿，向下传播所有变更 |
| 空项目启动失败 | 分析错误日志，修复后重新验证，**绝对不能跳过** |

## 最佳实践

1. **规则先行，代码后行**：先确认规则生效，再开始编码
2. **阶段确认，步步为营**：每阶段结束人工确认
3. **增量编码，单模块验证**：每次只开发一个模块
4. **错误反馈，精确描述**：把报错发给 AI，比描述问题更有效

## 人机分工

- **人类**：决策者、审核人、方向把控者
- **AI**：执行者、编码者、测试辅助者、文档生成者
