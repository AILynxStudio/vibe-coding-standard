# Vibe Coding 智能引导式 SKILL 深度设计方案

## 一、设计原则

本方案深度融入Vibe Coding标准体系的**六大核心精髓**：

1. **四层解耦架构**：规则优先级清晰，上层约束下层
2. **7条铁律强制执行**：四维拦截机制，不只是检查而是强制
3. **工具包组合逻辑**：根据场景自动选择最佳组合
4. **案例实战智慧**：将最佳实践融入引导流程
5. **质量保障闭环**：每阶段有门禁，全程可追溯
6. **7种场景深度适配**：不同场景有不同的执行策略

## 三、四层架构引擎设计

### 3.1 架构优先级执行逻辑

**优先级判断**：当规则冲突时，按以下优先级执行：
1. 项目硬约束层（最高优先级）
2. 通用流程Skill层
3. 技术规范层
4. 场景兜底层（最低优先级）

**执行流程**：
```
用户启动SKILL
    ↓
加载第四层：场景兜底规则（基础保障）
    ↓
加载第三层：技术规范（按需加载）
    ↓
加载第二层：通用流程Skill（8阶段驱动）
    ↓
加载第一层：项目硬约束（最高强制）
    ↓
四层规则融合，生成完整约束集
```

### 3.2 各层执行细节

**第一层：项目硬约束层执行**
- 启动时：检测是否存在 .cursorrules / CLAUDE.md，如不存在则引导创建
- 执行时：AI生成代码前检查是否违反硬约束，违反时主动拒绝执行
- 硬约束清单：目录结构、依赖管理、开发行为、Git规范、交付规范

**第二层：通用流程Skill层执行**
- 加载时机：用户启动SKILL时自动加载，每阶段开始时重新加载
- 执行方式：驱动8阶段串行执行，每阶段产出需人工确认

## 四、7条铁律强制执行引擎

### 4.1 四维拦截机制实现

**维度一：对AI（系统级拦截）**
- AI生成代码前，检查铁律清单
- 违反铁律时，拒绝生成并输出原因
- 提供符合铁律的替代方案

拦截规则示例：
- 铁律1：检测到批量生成多模块 → 阻断 → 提示"请单模块开发"
- 铁律2：检测到修改锁档文件 → 阻断 → 提示"锁档文件不可修改"
- 铁律3：检测到编造接口 → 阻断 → 提示"请先确认接口存在"
- 铁律6：空项目启动失败 → 阻断 → 提示"禁止编写业务代码"

**维度二：对人（流程级约束）**
- 每阶段结束，弹出确认清单
- 人工必须逐项确认
- 未确认则无法进入下一阶段

**维度三：对流程（审计级追踪）**
- 阶段2锁档文件提交Git
- 阶段4启动截图提交Git
- 每次代码提交附带阶段标记

Git提交规范：
- 阶段2：`[Stage-2] 锁档：流程图+目录树+依赖清单`
- 阶段4：`[Stage-4] 脚手架：空项目启动成功`
- 阶段5：`[Stage-5] 模块：{模块名称}开发完成`
- 阶段6：`[Stage-6] 测试：{模块名称}测试通过`

**维度四：对质量（门禁级控制）**
- 阶段6：异常测试覆盖率≥80%
- 阶段7：lint检查通过
- 阶段7：全量测试通过

### 4.2 铁律检查清单（每阶段执行）

**阶段1-2**：
- 铁律3：未编造不存在的接口
- 铁律7：未编写死代码、占位代码

**阶段3-4**：
- 铁律6：空项目初始化成功
- 铁律2：未修改阶段2锁档内容

**阶段5**：
- 铁律1：单模块开发，未批量生成
- 铁律3：未编造接口
- 铁律4：已实现loading/empty/error状态
- 铁律5：未修改已验收模块
- 铁律7：未编写死代码

**阶段6**：
- 铁律4：异常场景全覆盖
- 铁律5：修复未引入新Bug

**阶段7**：
- 铁律7：已清理冗余代码
- 铁律5：重构未破坏已有功能

## 五、工具包组合引擎

### 5.1 场景-工具包映射表

**智能IDE场景（Cursor、Windsurf、Trae）**
- 推荐组合：TK-1 + TK-2 + TK-5
- 部署方式：TK-1写入.cursorrules和CLAUDE.md，TK-2部署到.cursor/skills/vibe-coding-std/，TK-5作为Task模板
- 自动化程度：高（AI自动推进）

**插件增强场景（VSCode + AI插件）**
- 推荐组合：TK-1 + TK-2 + TK-5
- 部署方式：TK-1写入CLAUDE.md，TK-2部署到skills/vibe-coding-std/，TK-5作为Task模板人工驱动
- 自动化程度：中（人工驱动）

**桌面客户端场景（ChatGPT/Claude桌面版）**
- 推荐组合：TK-4 + TK-5
- 部署方式：TK-4每次对话开头粘贴，TK-5作为Task模板
- 自动化程度：低（全程手动）

**低代码Agent场景（Dify、Coze）**
- 推荐组合：TK-6
- 部署方式：TK-6固化为平台工作流
- 自动化程度：全自动

**无代码生成场景（v0、Bolt.new）**
- 推荐组合：无（脑中规划）
- 部署方式：阶段1-2脑中规划，阶段5增量追加，阶段6人工测试
- 自动化程度：零

**多智能体场景（AutoGen、AgentStack）**
- 推荐组合：TK-7 + TK-5
- 部署方式：TK-7分发给各Agent，TK-5作为统一Task模板
- 自动化程度：高（并行处理）

**企业级场景（Sourcegraph Cody）**
- 推荐组合：TK-1(组织级) + TK-5(CI化)
- 部署方式：TK-1写入组织级规则，TK-5集成到CI/CD
- 自动化程度：高（CI驱动）

### 5.2 工具包依赖关系

**核心依赖链**：
```
TK-1（项目硬约束）→ TK-2（通用流程）→ TK-5（任务模板）
     ↓                    ↓                    ↓
  规则定义            流程驱动            执行跟踪
```

**依赖规则**：
1. TK-1依赖TK-2：项目硬约束引用通用流程路径
2. TK-5依赖TK-1/TK-2：Task模板基于8阶段流程
3. TK-6依赖TK-2：低代码工作流固化8阶段
4. TK-7依赖TK-2：多智能体分工基于8阶段

**自动组合逻辑**：用户选择场景 → 自动推荐工具包组合 → 引导部署

## 六、8阶段深度引导设计

### 6.1 阶段1：需求收敛（融入案例智慧）

**AI引导话术**：
```
让我们从需求开始。我会问您几个问题来帮助梳理需求。

基于Vibe Coding的最佳实践，我需要了解：

1. **项目目标**：这个项目要解决什么问题？
2. **目标用户**：谁会使用这个项目？
3. **核心功能**：最核心的功能是什么？（最多3个）
4. **次要功能**：有哪些次要功能？
5. **扩展功能**：有哪些扩展功能？（可选）
6. **项目体量**：小型/中型/大型？
7. **运行载体**：Web/移动端/桌面/CLI？

请回答这些问题，我会帮您整理成需求文档。
```

**产出**：
- 极简项目简介（<=100字）
- 功能分级清单（核心/次要/扩展）
- 项目体量评估
- 运行载体确认

**门禁检查**：
```
AI: 我已经整理了您的需求：
[需求文档内容]

请确认：
- [ ] 项目目标清晰
- [ ] 功能分级合理
- [ ] 项目体量评估准确
- [ ] 运行载体确认

请确认是否准确？如有需要修改的地方请告诉我。
```

## 七、7种场景深度适配

### 7.1 场景一：智能IDE类（Cursor、Windsurf、Trae）

**特点**：Agent自主推进，自动化程度最高

**执行要点**：
- 阶段2：锁档后立即写入.cursorrules红线
- 阶段4：Agent执行命令，人工确认空项目启动成功
- 阶段5：给极精确指令，如"仅开发UserLogin组件的前端部分"
- 阶段6：要求AI生成全场景测试并在终端运行

**风险补救**：开启所有"需人工批准"选项，指令加严格限定词

### 7.2 场景二：插件增强类（VS Code + AI插件）

**特点**：与现有工作流集成，完全依赖人工驱动

**执行要点**：
- 阶段切换完全由人工下令
- 阶段4由人工在终端执行，成功后再告知AI
- 阶段5人工手动提供精确上下文，一次只给一个任务
- 阶段6人工运行测试，反馈报错给AI修复

**风险补救**：项目管理工具固化8阶段Task，每次对话首条粘贴规则

### 7.3 场景三：桌面AI客户端类

**特点**：开箱即用，无文件监听，全程手动

**执行要点**：
- 阶段1-3在客户端内对话完成，产出拷贝到本地
- 阶段4完全由人工本地执行，通过后告知AI
- 阶段5每次只描述一个模块，附上当前文件内容
- 阶段6-8人工主控

**风险补救**：每轮开头重述核心规则，用外部检查清单强制执行

### 7.4 场景四：低代码Agent平台类（Dify、Coze）

**特点**：全自动流转，灵活性低

**执行要点**：
- 将8阶段固化为平台工作流
- Agent必须串行执行，阶段4失败阻断
- 阶段5-6逐模块生成并自动测试

**风险补救**：提供受控回溯功能，定期更新技术栈Skill

### 7.5 场景五：无代码生成类（v0、Bolt.new）

**特点**：零门槛，极速生成，无法深度控制逻辑

**执行要点**：

- 阶段5改为增量追加功能
- 阶段6人工手动测试
- 阶段8一键部署或导出

**风险补救**：验证后导出代码转入场景一或二

### 7.6 场景六：多智能体协作类（AutoGen、AgentStack）

**特点**：并行处理，内建审查，通信可能出错

**执行要点**：
- 人类为项目经理，将阶段任务分派：
  - 阶段1-2：架构师Agent，人类审核锁定
  - 阶段3-4：DevOps Agent
  - 阶段5-6：多个程序员Agent并行，测试员Agent统一测试
  - 阶段7-8：集成Agent、文档Agent

### 7.7 场景七：企业级认知类（Sourcegraph Cody）

**特点**：组织级，输出符合内部规范

**执行要点**：

### 8.1 最佳实践

**增量开发原则**：
- 单次只开发一个最小功能模块
- 每个模块开发后立即本地运行验证
- 验证通过后立即提交Git

**人机协作原则**：
- 人类：决策者、审核人、方向把控者
- AI：执行者、编码者、测试辅助者、文档生成者
- 每阶段产出需人工确认

### 8.2 常见陷阱

**陷阱1：Cursor Agent一次生成多个文件**
- 解决方案：立即在对话中说"停止，你违反了单模块增量编码规则。请回滚到上一次提交，只保留[模块名]的改动。"

**陷阱2：npm create vite失败**
- 解决方案：检查Node.js版本是否≥18，终端输入`node -v`确认。版本过低则下载LTS版本。

**陷阱3：页面空白但没报错**
- 解决方案：检查App.tsx是否引入了组件，把App.tsx内容发给AI检查。

**陷阱4：AI编造不存在的接口**
- 解决方案：要求AI先输出接口清单，人工核实后再编码。

**陷阱5：阶段2锁档后需要修改**
- 解决方案：必须回到阶段2重新定稿，并向下传播所有变更。

### 8.3 案例洞察

## 十二、冲突解决机制设计

### 12.1 核心裁决规则

> **上层约束下层，规则出现歧义时以更高层级为准。**

这是整个冲突解决体系的唯一且根本的裁决规则。

### 12.2 四步判断法

```
步骤1: 判断是否有"铁律"被违反？
  ├── 是 → 立即停止执行，回滚，报告给人类 → END
  └── 否 → 进入步骤2

步骤2: 判断当前执行处于哪个阶段？
  ├── 阶段2已锁定的内容被改动？
  │   ├── 是 → 阻断，要求回到阶段2重新定稿 → END
  │   └── 否 → 进入步骤3
  └── 阶段4空项目初始化失败？
      ├── 是 → 绝对禁止进入阶段5 → END
      └── 否 → 进入步骤3

步骤3: 判断规则冲突来自哪个层级？
  ├── L1(硬约束) 与 L3(技术规范) 冲突 → L1 优先
  ├── L2(流程规则) 与 L3(技术规范) 冲突 → L2 优先
  ├── L1(硬约束) 与 L2(流程规则) 冲突 → L1 优先
  ├── L4(兜底) 与任意上层冲突 → 上层优先
  └── 同一层级内冲突 → 进入步骤4

步骤4: 无法自动裁决 → 报告给人类，由人类仲裁
```

### 12.3 冲突解决的四个层次

| 层次 | 机制 | 适用场景 | 工具支撑 |
|------|------|----------|----------|
| **预防层** | 场景隔离 + TK文件互斥 | 避免规则冲突发生 | case-guide对照表 |
| **检测层** | AI自检 + CI门禁 | 规则违反的即时发现 | CLAUDE.md + GitHub Actions |
| **裁决层** | 层级优先级 + 人类仲裁 | 规则歧义时的决策 | 阶段确认机制 |
| **恢复层** | Git回滚 + 阶段回溯 | 冲突发生后的修复 | Git + Task模板 |

### 12.4 冲突场景处理

**场景一：L1与L3的规范冲突**
- 表现：项目硬约束禁止某种依赖，但技术栈Skill推荐该依赖
- 裁决：L1胜出，AI不得引入未确认的依赖

**场景二：L2流程规则与L3技术实现的冲突**
- 表现：L2要求增量开发，但L3脚手架命令会一次生成多个文件
- 裁决：阶段4允许批量生成，阶段5必须回归增量模式

**场景三：L4兜底方案与上层规则的冲突**
- 表现：使用TK-3作为兜底时，无法实现L1/L2要求的本地文件监听
- 裁决：L4本身被设计为"咨询顾问"角色，从架构层面规避冲突

**场景四：多智能体间的技术规范冲突**
- 表现：多个AI Agent对同一接口产生不同理解
- 裁决：阶段2锁定的接口定义是"唯一真实来源"，人类拥有最终决定权

## 十三、铁律分层防御体系设计

### 13.1 五层防御结构

```
第一层：地基层 ──── 铁律6（地基必须稳固）
第二层：架构层 ──── 铁律2（架构不可动摇）
第三层：执行层 ──── 铁律1（增量推进） + 铁律3（接口真实）
第四层：质量层 ──── 铁律4（状态完整） + 铁律5（已验收锁定）
第五层：维护层 ──── 铁律7（代码纯净）
```

### 13.2 铁律依赖关系链

```
铁律6（空项目必须成功）
    └──► 铁律2（架构锁档不可变）
            └──► 铁律1（增量开发）
                    ├──► 铁律5（已验收模块不可改）
                    └──► 铁律3（禁止编造接口）
                            └──► 铁律4（完整状态处理）
                                    └──► 铁律7（禁止死代码）
```

### 13.3 绝对禁令vs条件禁令

**绝对禁令（零容忍，无例外）**：
- 铁律3：编造不存在的接口——涉及运行时安全
- 铁律6：空项目初始化失败就写业务代码——地基不牢

**条件禁令（有正式变更流程可豁免）**：
- 铁律2：可通过"需求变更回溯"机制正式修改架构
- 铁律5：可通过"需求变更单+重新验收"机制正式修改已验收模块

**执行禁令（有场景适配弹性）**：
- 铁律1：极简项目可简化为单模块
- 铁律4：无UI的纯逻辑模块可豁免状态处理
- 铁律7：第三方生成代码在阶段7统一清理

## 十四、隐性知识显性化设计

### 14.1 8个隐性知识及显性化方式

**隐性知识1：AI的"记忆"是上下文驱动的**
- 显性化：在案例中添加注释"AI的'记忆'就是当前对话的上下文。规则文件会被自动加载到上下文中，所以AI会'记住'这些规则。"

**隐性知识2：Git是"时间机器"**
- 显性化：在案例开头添加"Git不仅仅是版本控制，它是你的'时间机器'。每次提交都是一个存档点，出错时可以回到任意存档点。"

**隐性知识3：AI的"幻觉"需要人类审核**
- 显性化：在案例中添加警告"AI生成的内容可能包含错误或不符合预期的部分。请务必人工审核后再确认。"

**隐性知识4：规则文件的"优先级"机制**
- 显性化：在TK-1中添加注释"规则文件中的'最高强制规则'会被AI优先遵守，因为它是上下文中最早出现的约束。"

**隐性知识5：增量开发的"节奏感"**
- 显性化：在案例中添加经验总结"增量开发看似慢，实则快。因为每次只处理一个问题，错误定位成本最低，修复速度最快。"

**隐性知识6：AI的"上下文窗口"限制**
- 显性化：在案例中添加提示"AI的'记忆力'有限，对话太长会忘记前面的内容。所以每次只做一件事，做完就提交，避免上下文过长。"

**隐性知识7：错误信息是"最好的提示词"**
- 显性化：在案例中添加原则"错误信息是最好的提示词。与其描述'页面不好看'，不如把Console的红色报错复制给AI。"

**隐性知识8：规则文件是"AI的行为契约"**
- 显性化：在TK-1中添加解释"这份文件是给AI立规矩用的。它就像一份'行为契约'，AI会严格按照里面的规则执行。"

### 14.2 5个共性最佳实践及深层原因

**最佳实践1：规则先行，代码后行**
- 深层原因：AI的行为完全由上下文决定，规则文件是持久化的上下文

**最佳实践2：阶段确认，步步为营**
- 深层原因：这是人机协作的契约机制，没有确认AI会自动进入下一阶段

**最佳实践3：增量编码，单模块验证**
- 深层原因：AI的"幻觉"问题，生成的代码越多出错概率越高

**最佳实践4：测试清单，逐项验证**
- 深层原因：人类测试时容易遗漏边界情况，AI生成的测试清单是系统性的

**最佳实践5：错误反馈，精确描述**
- 深层原因：AI无法"猜测"错误，需要精确的错误信息

### 14.3 5个常见陷阱及深层原因

**陷阱1：AI跳过阶段直接生成代码**
- 深层原因：AI的训练数据中大量代码是一次性生成的，默认行为是"高效完成任务"

**陷阱2：AI批量生成多个文件**
- 深层原因：AI的上下文窗口有限，一次性生成所有代码可能超出限制

**陷阱3：页面空白但没报错**
- 深层原因：通常是组件引入失败或渲染逻辑错误，没有报错不代表没有错误

**陷阱4：npm create vite失败**
- 深层原因：Vite要求Node.js≥18，这是环境依赖问题，AI无法自动解决

**陷阱5：代码有重复逻辑**
- 深层原因：AI倾向于独立生成每个模块，不会主动识别重复

| **Dify/Coze** | ❌ 不需要 | 平台固化工作流 | 全自动 | TK-6 |
| **AutoGen/AgentStack** | ❌ 不需要 | 注入系统提示词 | 高 | TK-7 + TK-5 |

### 16.3 重点深入：为什么要区分？

**下载skill的本质**：将规则文件持久化到本地文件系统，AI工具自动监听并加载到上下文。

**不下载skill的局限**：只能通过手动粘贴提示词，规则无法持久化，每轮对话都可能丢失上下文。

**核心差异**：
```
下载skill：
- 规则持久化 → AI每次启动自动加载
- 上下文稳定 → 规则不会丢失
- 自动化高 → 人工介入少

不下载skill：
- 规则临时化 → 每轮对话需手动粘贴
- 上下文不稳定 → 规则可能丢失
- 自动化低 → 人工介入多
```

## 十八、skill定位与协作机制

### 18.1 本skill的定位

```
四层架构中的位置：

L1: 项目硬约束层 ──── .cursorrules / CLAUDE.md
L2: 通用流程Skill层 ── 本skill（vibe-coding-std）

L4: 场景兜底层 ──────── TK-3/TK-4/TK-6等

本skill定位：L2层（通用流程Skill层）
核心职责：驱动8阶段串行执行，每阶段产出需人工确认
```

### 18.2 与其他skill的协作关系

| 其他skill | 所属层级 | 协作方式 | 冲突处理 |
|-----------|:--------:|----------|----------|
| 项目硬约束（TK-1） | L1 | 本skill遵守L1的规则 | L1优先 |
| 技术栈Skill（React/Vue等） | L3 | 本skill调用L3的技术能力 | L2优先 |
| 场景兜底（TK-3/4/6） | L4 | 本skill在L4不可用时提供替代 | 上层优先 |
| 其他流程Skill | L2 | 人工选择激活哪个 | 人工仲裁 |

### 18.3 协作机制

**1. 层级优先级**
```
L1 > L2 > L3 > L4

本skill（L2）必须遵守L1的规则
本skill（L2）优先于L3的技术建议
本skill（L2）优先于L4的兜底方案
```

**2. 场景隔离**
```
不同场景使用不同的skill组合：
- 智能IDE场景：本skill + TK-1 + TK-5
- 桌面客户端场景：TK-4 + TK-5（不使用本skill）
- 低代码场景：TK-6（不使用本skill）
```

**3. 人类仲裁**
```
当skill之间冲突时：
- 自动裁决：上层约束下层
- 人工裁决：同一层级内冲突，报告给人类
- 最终决策：人类拥有最终决定权
```

### 18.4 本skill的能力边界

**能做的**：
- 引导用户执行8阶段流程
- 分析项目需求，推荐场景和工具包
- 协调子代理完成具体任务
- 检测规则冲突并裁决

**不能做的**：
- 直接修改其他skill的内容
- 保证其他skill的执行效果
- 自动安装缺失的skill
- 强制协调多个skill（需要人工仲裁）

**能力边界说明**：
```
本skill是一个"引导者"角色：
- 引导用户理解流程
- 引导AI遵守规则
- 引导项目按阶段推进

但最终决策权在人类：
- 阶段确认需要人工
- 冲突裁决需要人工
- 需求变更需要人工
```

## 二十、错误恢复机制设计

### 20.1 错误分类

| 错误类型 | 严重程度 | 恢复策略 |
|----------|:--------:|----------|
| **致命级** | ⭐⭐⭐⭐⭐ | 立即停止，回滚到安全状态 |
| **阻断级** | ⭐⭐⭐⭐ | 阻断当前阶段，修复后重试 |
| **警告级** | ⭐⭐⭐ | 记录警告，继续执行 |
| **建议级** | ⭐⭐ | 提供建议，不强制执行 |

### 20.2 错误恢复流程

```
检测到错误
    ↓
判断错误类型
    ├── 致命级 → 立即停止 → 回滚到上一次提交 → 报告给人类
    ├── 阻断级 → 阻断当前阶段 → 诊断原因 → 修复后重试
    ├── 警告级 → 记录警告 → 继续执行
    └── 建议级 → 提供建议 → 人工决定是否处理
```

### 20.3 具体错误恢复策略

#### 错误1：AI批量生成多模块（违反铁律1）

**检测方式**：AI一次生成多个文件
**恢复策略**：
```
1. 立即停止AI生成
2. 回滚到上一次提交：git reset --hard HEAD~1
3. 只保留目标模块的改动
4. 重新引导AI只生成单个模块
```

#### 错误2：AI编造不存在的接口（违反铁律3）

**检测方式**：接口调用失败，找不到定义
**恢复策略**：
```
1. 停止使用编造的接口
2. 要求AI先输出接口清单
3. 人工核实接口是否存在
4. 确认后才允许编码
```

#### 错误3：空项目初始化失败（违反铁律6）

**检测方式**：项目启动报错
**恢复策略**：
```
1. 阻断，禁止进入阶段5
2. 分析错误日志
3. 修复初始化问题
4. 重新验证空项目启动
5. 成功后才允许继续
```

#### 错误4：阶段2锁档后需要修改（违反铁律2）

**检测方式**：用户想要修改已锁定的内容
**恢复策略**：
```
1. 引导用户回到阶段2
2. 重新定稿架构设计
3. 向下传播所有变更
4. 重新执行阶段3-8
```

#### 错误5：测试覆盖率不足（违反铁律4）

**检测方式**：异常测试覆盖率<80%
**恢复策略**：
```
1. 阻断，禁止进入阶段7
2. 分析缺失的测试场景
3. 补充测试用例
4. 重新验证覆盖率
5. 达标后才允许继续
```

#### 错误6：lint检查失败

**检测方式**：代码风格不符合规范
**恢复策略**：
```
1. 阻断，禁止进入阶段8
2. 分析lint错误
3. 修复代码风格问题
4. 重新运行lint检查
5. 通过后才允许继续
```

## 二十一、版本管理策略设计

### 21.1 版本号规则

根据CONTRIBUTING.md定义：

|----------|-----------|----------|------|

| **次版本** | 0.X.0 | 新增案例、工具包微调 | 1.1.0 |
| **修订号** | 0.0.X | 文字修正、格式调整 | 1.0.1 |

### 21.2 版本升级路径

```
当前版本：1.0.0

升级路径：
├── 修订号升级（1.0.0 → 1.0.1）
│   └── 触发：文字修正、格式调整
│   └── 影响：无功能变更
│
├── 次版本升级（1.0.0 → 1.1.0）
│   └── 触发：新增案例、工具包微调
│   └── 影响：新增功能，向后兼容
│
└── 主版本升级（1.0.0 → 2.0.0）
    └── 触发：标准体系重大调整
    └── 影响：可能不向后兼容
```

### 21.3 版本管理流程

```
1. 修改提案
   └── 通过GitHub Issue或PR提出修改建议

2. 一致性检查
   └── 确保修改与标准体系保持一致

3. 版本号更新
   └── 根据变更类型更新相应版本号

4. 审核合并
   └── 项目维护者审核后合并到主分支

5. 发布更新
   └── 更新CHANGELOG，发布新版本
```

### 21.4 工具包版本管理

**工具包版本号**：
- TK-1：1.0.0
- TK-2：1.0.0
- TK-3：1.0.0
- TK-4：1.0.0
- TK-5：1.0.0
- TK-6：1.0.0
- TK-7：1.0.0

**工具包更新规则**：
```
1. 工具包内容变更 → 更新工具包版本号
2. 标准体系调整 → 更新所有受影响的工具包
3. 新增场景 → 新增对应的工具包
4. 废弃场景 → 标记工具包为废弃
```

### 21.5 向后兼容策略

**兼容性原则**：
```
1. 次版本升级必须向后兼容
2. 主版本升级可以不向后兼容
3. 废弃功能至少保留一个次版本
```

**升级指南**：
```
从1.0.0升级到1.1.0：
- 新增功能，无破坏性变更
- 直接替换文件即可

从1.0.0升级到2.0.0：
- 可能有破坏性变更
- 需要阅读迁移指南
- 可能需要修改配置
```

**自动检查**：
```
# 检查当前版本
cat VERSION

# 检查是否有更新
git fetch --tags
git describe --tags --abbrev=0
```

**手动检查**：
