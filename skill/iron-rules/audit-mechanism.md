# 审计追踪机制

## 概述

审计追踪机制负责记录开发过程中的所有变更，确保全程可追溯。

## 审计维度

| 维度 | 审计对象 | 审计方式 | 审计时机 |
|------|----------|----------|----------|
| 规则维度 | AI | CLAUDE.md系统级拦截 | 代码生成前 |
| 流程维度 | 人类 | Task模板逐项确认 | 阶段结束时 |
| 审计维度 | 过程 | Git提交记录 | 每次变更时 |
| 质量维度 | 代码 | CI门禁自动检查 | 代码提交时 |

## Git审计机制

### Git提交规范

**阶段标记**：
- 阶段2：`[Stage-2] 锁档：流程图+目录树+依赖清单`
- 阶段4：`[Stage-4] 脚手架：空项目启动成功`
- 阶段5：`[Stage-5] 模块：{模块名称}开发完成`
- 阶段6：`[Stage-6] 测试：{模块名称}测试通过`
- 阶段7：`[Stage-7] 重构：全局联调优化完成`
- 阶段8：`[Stage-8] 交付：项目完成，文档归档`

**提交示例**：
```bash
# 阶段2锁档
git add .
git commit -m "[Stage-2] 锁档：流程图+目录树+依赖清单"

# 阶段4脚手架
git add .
git commit -m "[Stage-4] 脚手架：空项目启动成功"

# 阶段5模块开发
git add .
git commit -m "[Stage-5] 模块：Header组件开发完成"

# 阶段6测试验证
git add .
git commit -m "[Stage-6] 测试：Header组件测试通过"

# 阶段7重构优化
git add .
git commit -m "[Stage-7] 重构：全局联调优化完成"

# 阶段8交付归档
git add .
git commit -m "[Stage-8] 交付：项目完成，文档归档"
```

### Git审计内容

**阶段2锁档审计**：
```
审计内容：
- Mermaid业务流程图
- 固定目录树
- 依赖清单
- 接口定义

审计方式：
- 锁档文件提交Git
- 版本控制可追溯
- 变更需重新定稿
```

**阶段4脚手架审计**：
```
审计内容：
- 项目初始化命令
- 依赖安装记录
- 空项目启动截图

审计方式：
- 启动截图提交Git
- 成功标志可验证
- 失败记录可追溯
```

**阶段5模块开发审计**：
```
审计内容：
- 模块代码变更
- 模块测试结果
- 模块验收确认

审计方式：
- 每个模块单独提交
- 提交信息包含模块名称
- 变更记录可追溯
```

**阶段6测试验证审计**：
```
审计内容：
- 测试清单
- 测试结果
- 覆盖率报告

审计方式：
- 测试结果提交Git
- 覆盖率报告可验证
- 失败记录可追溯
```

### Task模板审计流程

```
阶段完成
    ↓
AI生成Task模板
    ↓
用户逐项确认
    ├── 全部确认 → 记录确认人和时间 → 进入下一阶段
    └── 部分未确认 → 补充确认 → 重新检查
```

## CI门禁审计机制（企业级）

### CI门禁规则

```yaml
# .github/workflows/vibe-coding-audit.yml
name: Vibe Coding Audit

on: [push, pull_request]

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: 检查铁律1：禁止批量生成
        run: |
          # 检查单次提交文件数量
          if [ $(git diff --name-only HEAD~1 HEAD | wc -l) -gt 5 ]; then
            echo "❌ 单次提交文件数量过多，请单模块开发"
            exit 1
          fi
      
      - name: 检查铁律2：禁止修改锁档
        run: |
          # 检查是否修改锁档文件
          if git diff --name-only HEAD~1 HEAD | grep -q "architecture\|dependencies"; then
            echo "❌ 检测到修改锁档文件，请回到阶段2重新定稿"
            exit 1
          fi
      
      - name: 检查铁律3：禁止编造接口
        run: |
          # 检查API调用是否存在于接口定义中
          # 自定义检查逻辑
      
      - name: 检查铁律6：空项目必须成功
        run: |
          # 检查项目是否能成功启动
          npm install && npm run build
      
      - name: 检查测试覆盖率
        run: |
          # 检查测试覆盖率是否≥80%
          npm run test:coverage
          if [ $(cat coverage/coverage-summary.json | jq '.total.lines.pct') -lt 80 ]; then
            echo "❌ 测试覆盖率不足80%"
            exit 1
          fi
      
      - name: 检查代码风格
        run: |
          # 检查lint是否通过
          npm run lint
```

### CI门禁审计内容

| 检查项 | 检查方式 | 门禁条件 |
|--------|----------|----------|
| 铁律1 | 检查单次提交文件数量 | ≤5个文件 |
| 铁律2 | 检查是否修改锁档文件 | 不可修改 |
| 铁律3 | 检查API调用是否存在于接口定义 | 必须存在 |
| 铁律6 | 检查项目是否能成功启动 | 必须成功 |
| 测试覆盖率 | 检查测试覆盖率报告 | ≥80% |
| 代码风格 | 检查lint结果 | 必须通过 |
