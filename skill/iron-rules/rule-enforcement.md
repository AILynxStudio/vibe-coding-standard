# 铁律强制执行逻辑

## 概述

铁律强制执行逻辑负责将7条铁律转化为可执行的规则，并在开发过程中强制执行。

## 铁律执行矩阵

| 铁律 | 执行对象 | 执行方式 | 检查时机 | 处理方式 |
|------|----------|----------|----------|----------|
| 铁律1 | AI | 系统级拦截 | 阶段5 | 阻断+回滚 |
| 铁律2 | AI+人 | 规则文件+人工确认 | 阶段2-8 | 阻断+回溯 |
| 铁律3 | AI | 系统级拦截 | 所有阶段 | 阻断+拒绝 |
| 铁律4 | AI+人 | 规则文件+测试清单 | 阶段6 | 阻断+补充 |
| 铁律5 | AI+人 | 规则文件+人工确认 | 阶段5-8 | 阻断+回滚 |
| 铁律6 | AI+人 | 硬性门槛 | 阶段4 | 阻断+修复 |
| 铁律7 | AI | 规则文件+lint检查 | 阶段7 | 阻断+清理 |

## 铁律1：禁止批量生成多模块

### 执行逻辑

```
AI生成代码前
    ↓
检查是否批量生成多模块？
    ├── 是 → 阻断 → 提示"请单模块开发" → END
    └── 否 → 允许生成
```

### 拦截规则

- 检测到一次生成多个文件 → 阻断
- 检测到全量代码生成 → 阻断

### 恢复策略

```
1. 立即停止AI生成
2. 回滚到上一次提交：git reset --hard HEAD~1
3. 只保留目标模块的改动
4. 重新引导AI只生成单个模块
```

## 铁律2：阶段2定稿不得私自修改

### 执行逻辑

```
AI修改文件前
    ↓
检查是否修改阶段2锁档内容？
    ├── 是 → 阻断 → 提示"锁档文件不可修改" → END
    └── 否 → 允许修改
```

### 拦截规则

- 检测到修改目录结构 → 阻断
- 检测到修改依赖清单 → 阻断
- 检测到修改接口定义 → 阻断

### 恢复策略

```
1. 引导用户回到阶段2
2. 重新定稿架构设计
3. 向下传播所有变更
4. 重新执行阶段3-8
```

## 铁律3：严禁编造不存在的接口

### 执行逻辑

```
AI生成代码前
    ↓
检查是否编造接口？
    ├── 是 → 阻断 → 提示"请先确认接口存在" → END
    └── 否 → 允许生成
```

### 拦截规则

- 检测到不存在的API调用 → 阻断
- 检测到不存在的函数调用 → 阻断
- 检测到不存在的依赖包 → 阻断

### 恢复策略

```
1. 停止使用编造的接口
2. 要求AI先输出接口清单
3. 人工核实接口是否存在
4. 确认后才允许编码
```

## 铁律4：必须补充完整状态处理

### 执行逻辑

```
AI生成代码后
    ↓
检查是否包含完整状态处理？
    ├── 否 → 阻断 → 提示"请补充loading/empty/error状态" → END
    └── 是 → 允许继续
```

### 拦截规则

- 网络请求缺少loading状态 → 阻断
- 数据展示缺少empty状态 → 阻断
- 用户输入缺少异常处理 → 阻断

### 恢复策略

```
1. 分析缺失的状态处理
2. 补充loading/empty/error状态
3. 重新验证
```

## 铁律5：已验收模块禁止随意修改

### 执行逻辑

```
AI修改文件前
    ↓
检查是否修改已验收模块？
    ├── 是 → 阻断 → 提示"已验收模块不可修改" → END
    └── 否 → 允许修改
```

### 拦截规则

- 检测到修改已测试通过的模块 → 阻断
- 检测到修改已提交Git的模块 → 阻断

### 恢复策略

```
1. 评估修改的必要性
2. 如有必要，走正式变更流程
3. 需求变更单 → 人工审批 → 重新验收
```

## 铁律6：空项目初始化失败禁止编写业务代码

### 执行逻辑

```
阶段4完成
    ↓
空项目是否成功启动？
    ├── 否 → 绝对禁止进入阶段5 → END
    └── 是 → 允许进入阶段5
```

### 拦截规则

- 空项目启动报错 → 阻断
- 依赖安装失败 → 阻断
- 环境配置错误 → 阻断

### 恢复策略

```
1. 分析错误日志
2. 修复初始化问题
3. 重新验证空项目启动
4. 成功后才允许继续
```

## 铁律7：禁止编写死代码、占位代码

### 执行逻辑

```
AI生成代码后
    ↓
检查是否包含死代码？
    ├── 是 → 阻断 → 提示"请清理死代码" → END
    └── 否 → 允许继续
```

### 拦截规则

- 检测到TODO注释 → 警告
- 检测到空函数体 → 阻断
- 检测到注释掉的代码 → 阻断
- 检测到永远不会执行的分支 → 阻断

### 恢复策略

```
1. 识别死代码类型
2. 清理或实现必要代码
3. 重新验证
```

## 执行保障机制

### 对AI的执行保障

1. **规则文件**：铁律写入.cursorrules/CLAUDE.md
2. **系统级拦截**：AI生成代码前自动检查
3. **拒绝执行**：违反铁律时主动拒绝

### 对人的执行保障

1. **Task模板**：8阶段做成Task模板
2. **逐项确认**：每阶段结束人工确认
3. **不可跳过**：未确认无法进入下一阶段

### 对流程的执行保障

1. **Git审计**：锁档文件、启动截图提交Git
2. **版本控制**：所有变更可追溯
3. **回滚机制**：出错时可回滚到安全状态

### 对质量的执行保障

1. **测试覆盖率**：阶段6异常测试覆盖率≥80%
2. **lint检查**：阶段7代码风格检查
3. **CI门禁**：企业级场景自动化门禁
