# 常见陷阱

## 概述

常见陷阱总结了Vibe Coding标准体系中的常见问题和解决方案。

## 陷阱1：AI跳过阶段直接生成代码

### 表现

AI跳过阶段1-2，直接生成代码

### 深层原因

- AI的训练数据中，大量代码是一次性生成的
- AI的默认行为是"高效完成任务"，而非"遵循流程"
- 没有规则文件约束时，AI会使用最短路径

### 解决方案

```
1. 在对话中指出："你跳过了阶段X"
2. 要求AI回到指定阶段
3. 按流程重新执行
```

### 预防措施

```
1. 部署规则文件
2. 明确阶段要求
3. 人工确认阶段完成
```

## 陷阱2：AI批量生成多个文件

### 表现

AI一次生成多个文件

### 深层原因

- AI的上下文窗口有限
- 一次性生成所有代码可能超出限制
- 批量生成的代码耦合度高

### 解决方案

```
1. 立即停止AI生成
2. 回滚到上一次提交：git reset --hard HEAD~1
3. 只保留目标模块的改动
4. 重新引导AI只生成单个模块
```

### 预防措施

```
1. 指令加严格限定词
2. 开启"需人工批准"选项
3. 使用Task模板逐项确认
```

## 陷阱3：页面空白但没报错

### 表现

页面空白，控制台无报错

### 深层原因

- 通常是组件引入失败
- 或渲染逻辑错误
- 没有报错不代表没有错误

### 解决方案

```
1. 检查组件引入
2. 检查渲染逻辑
3. 把App.tsx内容发给AI检查
```

### 预防措施

```
1. 增量开发，逐步验证
2. 每个模块单独测试
3. 使用React DevTools调试
```

## 陷阱4：npm create vite失败

### 表现

npm create vite命令失败

### 深层原因

- Vite要求Node.js≥18
- 这是环境依赖问题
- AI无法自动解决

### 解决方案

```
1. 检查Node.js版本：node -v
2. 如果版本过低，下载LTS版本
3. 重新运行初始化命令
```

### 预防措施

```
1. 提前检查环境要求
2. 使用nvm管理Node版本
3. 在阶段3确认环境就绪
```

## 陷阱5：代码有重复逻辑

### 表现

代码中存在重复逻辑

### 深层原因

- AI倾向于独立生成每个模块
- 不会主动识别重复
- 重复代码是技术债务

### 解决方案

```
1. 在阶段7检查重复逻辑
2. 提取公共函数/组件
3. 替换重复代码
```

### 预防措施

```
1. 阶段2设计清晰的模块划分
2. 阶段5注意代码复用
3. 阶段7全局重构
```

## 陷阱6：AI编造不存在的接口

### 表现

AI调用不存在的API或函数

### 深层原因

- AI幻觉问题
- AI自信地编造接口
- 编造的接口可能引入漏洞

### 解决方案

```
1. 要求AI先输出接口清单
2. 人工核实接口是否存在
3. 确认后才允许编码
```

### 预防措施

```
1. 阶段2锁定接口定义
2. 使用强类型接口契约
3. 人工审核关键接口
```

## 陷阱7：阶段2锁档后需要修改

### 表现

阶段2锁定后需要修改内容

### 深层原因

- 需求变更
- 架构设计不合理
- 技术选型错误

### 解决方案

```
1. 记录变更需求
2. 回到阶段2重新定稿
3. 向下传播所有变更
4. 重新执行后续阶段
```

### 预防措施

```
1. 阶段1充分沟通需求
2. 阶段2仔细设计架构
3. 人工确认后锁定
```

## 陷阱8：测试覆盖率不足

### 表现

测试覆盖率<80%

### 深层原因

- 测试场景不全面
- 边界条件遗漏
- 异常场景未覆盖

### 解决方案

```
1. 分析缺失的测试场景
2. 补充测试用例
3. 重新测试
4. 更新覆盖率报告
```

### 预防措施

```
1. 使用测试清单
2. 覆盖正常/异常/边界场景
3. 自动化测试
```

## 陷阱9：lint检查失败

### 表现

代码风格不符合规范

### 深层原因

- 代码风格不统一
- 命名不规范
- 格式不一致

### 解决方案

```
1. 运行lint检查
2. 分析错误
3. 修复代码风格
4. 重新检查
```

### 预防措施

```
1. 使用ESLint + Prettier
2. 配置统一的代码风格
3. 提交前自动检查
```

## 陷阱10：已验收模块被修改

### 表现

已测试通过的模块被修改

### 深层原因

- AI"顺手"优化未让它动的模块
- 修复Bug时引入新Bug
- 回归风险

### 解决方案

```
1. 评估修改的必要性
2. 如有必要，走正式变更流程
3. 需求变更单 → 人工审批 → 重新验收
```

### 预防措施

```
1. 明确模块边界
2. 已验收模块标记为只读
3. 修改前人工确认
```
