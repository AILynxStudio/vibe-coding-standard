# 案例洞察

## 概述

案例洞察总结了从实战案例中提取的深层经验和隐性知识。

## 隐性知识

### 1. AI的"记忆"是上下文驱动的

**案例体现**：
- case-03阶段3要求AI"复述规则"
- case-06中多个Agent共享TK-7的通用规则

**深层原因**：
- AI的"记忆"就是当前对话的上下文
- 规则文件会被自动加载到上下文中
- 所以AI会"记住"这些规则

**显性化方式**：
```
在案例中添加注释：
"AI的'记忆'就是当前对话的上下文。规则文件会被自动加载到上下文中，所以AI会'记住'这些规则。"
```

### 2. Git是"时间机器"

**案例体现**：
- case-03每个模块后都要求"提交Git"
- FAQ中建议"回滚到上一次提交"

**深层原因**：
- Git不仅仅是版本控制
- 它是你的"时间机器"
- 每次提交都是一个存档点

**显性化方式**：
```
在案例开头添加：
"Git不仅仅是版本控制，它是你的'时间机器'。每次提交都是一个存档点，出错时可以回到任意存档点。"
```

### 3. AI的"幻觉"需要人类审核

**案例体现**：
- 所有案例都强调"AI产出参考（需人工审核）"
- case-03阶段1的产出标注为"需人工审核"

**深层原因**：
- AI会自信地编造内容
- 生成的内容可能包含错误
- 人类审核是最后的防线

**显性化方式**：
```
在案例中添加警告：
"AI生成的内容可能包含错误或不符合预期的部分。请务必人工审核后再确认。"
```

### 4. 规则文件的"优先级"机制

**案例体现**：
- case-03步骤4验证"AI是否能复述最高优先级规则"
- TK-1中定义了"项目最高强制规则"

**深层原因**：
- 规则文件中的"最高强制规则"会被AI优先遵守
- 因为它是上下文中最早出现的约束
- 优先级机制是隐式的

**显性化方式**：
```
在TK-1中添加注释：
"规则文件中的'最高强制规则'会被AI优先遵守，因为它是上下文中最早出现的约束。"
```

### 5. 增量开发的"节奏感"

**案例体现**：
- case-03的模块拆分有严格顺序
- 每个模块后都有"验证→提交→继续"的循环

**深层原因**：
- 增量开发看似慢，实则快
- 因为每次只处理一个问题
- 错误定位成本最低，修复速度最快

**显性化方式**：
```
在案例中添加经验总结：
"增量开发看似慢，实则快。因为每次只处理一个问题，错误定位成本最低，修复速度最快。"
```

### 6. AI的"上下文窗口"限制

**案例体现**：
- case-06中提到"降低max_round值"
- case-02中每次只做一件事

**深层原因**：
- AI的"记忆力"有限
- 对话太长会忘记前面的内容
- 所以每次只做一件事

**显性化方式**：
```
在案例中添加提示：
"AI的'记忆力'有限，对话太长会忘记前面的内容。所以每次只做一件事，做完就提交，避免上下文过长。"
```

### 7. 错误信息是"最好的提示词"

**案例体现**：
- 所有FAQ都强调"把报错发给AI"
- case-02: "截图或复制错误文字发给AI"

**深层原因**：
- 错误信息是最好的提示词
- 与其描述"页面不好看"
- 不如把Console的红色报错复制给AI

**显性化方式**：
```
在案例中添加原则：
"错误信息是最好的提示词。与其描述'页面不好看'，不如把Console的红色报错复制给AI。"
```

### 8. 规则文件是"AI的行为契约"

**案例体现**：
- TK-1定义了"项目最高强制规则"
- case-03步骤4验证AI是否遵守规则

**深层原因**：
- 这份文件是给AI立规矩用的
- 它就像一份"行为契约"
- AI会严格按照里面的规则执行

**显性化方式**：
```
在TK-1中添加解释：
"这份文件是给AI立规矩用的。它就像一份'行为契约'，AI会严格按照里面的规则执行。"
```

## 共性最佳实践

### 1. 规则先行，代码后行

**案例体现**：
- 所有案例第一步都是"部署规则文件"

**深层原因**：
- AI的行为完全由上下文决定
- 规则文件是持久化的上下文
- 没有规则文件，AI会使用默认行为

### 2. 阶段确认，步步为营

**案例体现**：
- 所有案例都遵循"阶段X确认，进入阶段Y"

**深层原因**：
- 这是人机协作的契约机制
- 没有确认，AI会自动进入下一阶段
- 确认语句本身也是上下文的一部分

### 3. 增量编码，单模块验证

**案例体现**：
- 所有编码案例都强调"每次只让AI做一件事"

**深层原因**：
- AI的"幻觉"问题
- 生成的代码越多，出错概率越高
- 增量编码使得错误定位成本最低

### 4. 测试清单，逐项验证

**案例体现**：
- case-02、case-03、case-04、case-07都要求AI生成测试清单

**深层原因**：
- 人类测试时容易遗漏边界情况
- AI生成的测试清单是系统性的
- 逐项测试的过程本身就是对需求的二次确认

### 5. 错误反馈，精确描述

**案例体现**：
- 所有案例的FAQ都强调"把完整报错发给AI"

**深层原因**：
- AI无法"猜测"错误
- 需要精确的错误信息
- 精确的错误描述能让AI一次性修复

## 常见陷阱

### 1. AI跳过阶段直接生成代码

**案例体现**：
- case-05: "Agent 跳过阶段直接生成代码？"

**深层原因**：
- AI的默认行为是"高效完成任务"
- 没有规则文件约束时，AI会使用最短路径

### 2. AI批量生成多个文件

**案例体现**：
- case-03: "Cursor Agent 一次生成多个文件怎么办？"

**深层原因**：
- AI的上下文窗口有限
- 批量生成的代码耦合度高

### 3. 页面空白但没报错

**案例体现**：
- case-03: "页面空白但没报错？"

**深层原因**：
- 通常是组件引入失败
- 或渲染逻辑错误

### 4. npm create vite失败

**案例体现**：
- case-03: "npm create vite 失败？"

**深层原因**：
- Vite要求Node.js≥18
- 这是环境依赖问题

### 5. 代码有重复逻辑

**案例体现**：
- case-05: "生成的代码有重复？"

**深层原因**：
- AI倾向于独立生成每个模块
- 不会主动识别重复
