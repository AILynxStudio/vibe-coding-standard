# 场景适配引擎

## 概述

场景适配引擎负责根据用户的开发工具和项目需求，自动选择合适的场景和工具包组合。

## 7种场景概览

| 场景 | 名称 | 能力等级 | 适用项目 | 核心优势 | 核心限制 |
|------|------|----------|----------|----------|----------|
| 场景一 | 智能IDE类 | 顶级 | 复杂全栈应用 | Agent自主推进 | 规则错误导致失控 |
| 场景二 | 插件增强类 | 高级 | 日常开发、中小型项目 | 与现有工作流集成 | 完全依赖人工驱动 |
| 场景三 | 桌面AI客户端类 | 中级 | 快速原型、中型项目 | 开箱即用 | 无文件监听，全程手动 |
| 场景四 | 低代码Agent平台类 | 自动化 | 标准化模板、批量表单 | 全自动流转 | 灵活性低 |
| 场景五 | 无代码生成类 | 非工程级 | 原型、内部小工具 | 零门槛，极速生成 | 无法深度控制逻辑 |
| 场景六 | 多智能体协作类 | 实验-高级 | 复杂系统、并行子任务 | 并行处理，内建审查 | 通信可能出错，成本高 |
| 场景七 | 企业级认知类 | 组织级 | 多团队大型系统 | 输出符合内部规范 | 建设知识索引投入大 |

## 各场景详细适配

### 场景一：智能IDE类（Cursor、Windsurf、Trae）

**特点**：Agent自主推进，自动化程度最高

**工具包组合**：TK-1 + TK-2 + TK-5

**部署方式**：
1. TK-1 写入 .cursorrules 和 CLAUDE.md
2. TK-2 部署到 .cursor/skills/vibe-coding-std/
3. TK-5 作为Task模板使用

**执行要点**：
- 阶段2：锁档后立即写入.cursorrules红线
- 阶段4：Agent执行命令，人工确认空项目启动成功
- 阶段5：给极精确指令，如"仅开发UserLogin组件的前端部分"
- 阶段6：要求AI生成全场景测试并在终端运行

**风险补救**：开启所有"需人工批准"选项，指令加严格限定词

### 场景二：插件增强类（VS Code + AI插件）

**特点**：与现有工作流集成，完全依赖人工驱动

**工具包组合**：TK-1 + TK-2 + TK-5

**部署方式**：
1. TK-1 写入 CLAUDE.md
2. TK-2 部署到 skills/vibe-coding-std/
3. TK-5 作为Task模板，人工驱动

**执行要点**：
- 阶段切换完全由人工下令
- 阶段4由人工在终端执行，成功后再告知AI
- 阶段5人工手动提供精确上下文，一次只给一个任务
- 阶段6人工运行测试，反馈报错给AI修复

**风险补救**：项目管理工具固化8阶段Task，每次对话首条粘贴规则

### 场景三：桌面AI客户端类

**特点**：开箱即用，无文件监听，全程手动

**工具包组合**：TK-4 + TK-5

**部署方式**：
1. TK-4 每次对话开头粘贴
2. TK-5 作为Task模板

**执行要点**：
- 阶段1-3在客户端内对话完成，产出拷贝到本地
- 阶段4完全由人工本地执行，通过后告知AI
- 阶段5每次只描述一个模块，附上当前文件内容
- 阶段6-8人工主控

**风险补救**：每轮开头重述核心规则，用外部检查清单强制执行

### 场景四：低代码Agent平台类（Dify、Coze）

**特点**：全自动流转，灵活性低

**工具包组合**：TK-6

**部署方式**：
1. TK-6 固化为平台工作流

**执行要点**：
- 将8阶段固化为平台工作流
- Agent必须串行执行，阶段4失败阻断
- 阶段5-6逐模块生成并自动测试

**风险补救**：提供受控回溯功能，定期更新技术栈Skill

### 场景五：无代码生成类（v0、Bolt.new）

**特点**：零门槛，极速生成，无法深度控制逻辑

**工具包组合**：TK-3（仅咨询）

**部署方式**：
1. 复制TK-3内容作为系统提示词

**执行要点**：
- 阶段1-2在脑中规划
- 阶段5改为增量追加功能
- 阶段6人工手动测试
- 阶段8一键部署或导出

**风险补救**：验证后导出代码转入场景一或二

### 场景六：多智能体协作类（AutoGen、AgentStack）

**特点**：并行处理，内建审查，通信可能出错

**工具包组合**：TK-7 + TK-5

**部署方式**：
1. TK-7 分发给各Agent
2. TK-5 作为统一Task模板

**执行要点**：
- 人类为项目经理，将阶段任务分派：
  - 阶段1-2：架构师Agent，人类审核锁定
  - 阶段3-4：DevOps Agent
  - 阶段5-6：多个程序员Agent并行，测试员Agent统一测试
  - 阶段7-8：集成Agent、文档Agent

**风险补救**：强类型接口契约，仅在阶段5-6启用多Agent并行

### 场景七：企业级认知类（Sourcegraph Cody）

**特点**：组织级，输出符合内部规范

**工具包组合**：TK-1(组织级) + TK-5(CI化)

**部署方式**：
1. TK-1 写入组织级规则
2. TK-5 集成到CI/CD

**执行要点**：
- 阶段2锁档需通过架构评审委员会
- 阶段4空项目需通过安全扫描
- 阶段5-6提交代码必须通过CI门禁
- 阶段7利用Cody批量修改功能安全重构

**风险补救**：设立紧急Hotfix流程，从核心项目试点

## 工具包组合推荐

### 核心依赖链

```
TK-1（项目硬约束）→ TK-2（通用流程）→ TK-5（任务模板）
     ↓                    ↓                    ↓
  规则定义            流程驱动            执行跟踪
```

### 场景-工具包映射表

| 场景 | 推荐组合 | 部署方式 | 自动化程度 |
|------|----------|----------|------------|
| 智能IDE | TK-1 + TK-2 + TK-5 | 文件部署 | 高 |
| 插件增强 | TK-1 + TK-2 + TK-5 | 文件部署+手动加载 | 中 |
| 桌面客户端 | TK-4 + TK-5 | 每次对话粘贴 | 低 |
| 低代码平台 | TK-6 | 平台固化 | 全自动 |
| 无代码生成 | TK-3 | 系统提示词 | 无 |
| 多智能体 | TK-7 + TK-5 | 角色分发 | 高 |
| 企业级 | TK-1(组织级) + TK-5(CI化) | CI/CD集成 | 高 |

## 场景升级路径

```
原型 → 正式产品
├── 场景五（无代码）→ 场景三（桌面客户端）→ 场景一（智能IDE）
└── 场景五（无代码）→ 场景二（插件增强）→ 场景一（智能IDE）

个人 → 团队 → 企业
├── 场景一（智能IDE）→ 场景六（多智能体）→ 场景七（企业级）
└── 场景二（插件增强）→ 场景六（多智能体）→ 场景七（企业级）
```
