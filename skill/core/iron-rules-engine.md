# 铁律执行引擎

## 概述

铁律执行引擎负责强制执行7条铁律，通过四维拦截机制确保代码质量。

## 7条铁律清单

| 铁律 | 内容 | 对抗问题 |
|------|------|----------|
| **铁律1** | 禁止批量生成多模块、禁止全量代码一次性生成 | AI惰性 & 熵增失控 |
| **铁律2** | 阶段2定稿的目录、依赖、接口不得私自修改 | 架构随意变动 |
| **铁律3** | 严禁编造不存在的接口、函数、字段、第三方SDK | AI幻觉 |
| **铁律4** | 网络请求、用户输入等必须补充加载状态、空数据状态、异常捕获与错误提示 | 只写"快乐路径" |
| **铁律5** | 已验收通过的稳定模块，无正式需求变更禁止随意修改 | 修复引入新Bug |
| **铁律6** | 空项目初始化失败，绝对不允许编写业务代码 | 在错误基础上建造 |
| **铁律7** | 禁止编写死代码、占位代码、冗余无效代码 | 代码腐烂 |

## 五层防御结构

```
第一层：地基层 ──── 铁律6（地基必须稳固）
第二层：架构层 ──── 铁律2（架构不可动摇）
第三层：执行层 ──── 铁律1（增量推进） + 铁律3（接口真实）
第四层：质量层 ──── 铁律4（状态完整） + 铁律5（已验收锁定）
第五层：维护层 ──── 铁律7（代码纯净）
```

## 铁律依赖关系链

```
铁律6（空项目必须成功）
    └──► 铁律2（架构锁档不可变）
            └──► 铁律1（增量开发）
                    ├──► 铁律5（已验收模块不可改）
                    └──► 铁律3（禁止编造接口）
                            └──► 铁律4（完整状态处理）
                                    └──► 铁律7（禁止死代码）
```

## 四维拦截机制

### 维度一：对AI（系统级拦截）

**执行方式**：铁律写入 `.cursorrules` / `CLAUDE.md`

**拦截逻辑**：
1. AI生成代码前，检查铁律清单
2. 违反铁律时，拒绝生成并输出原因
3. 提供符合铁律的替代方案

**拦截规则示例**：
- 铁律1拦截：检测到批量生成多模块 → 阻断 → 提示"请单模块开发"
- 铁律2拦截：检测到修改锁档文件 → 阻断 → 提示"锁档文件不可修改"
- 铁律3拦截：检测到编造接口 → 阻断 → 提示"请先确认接口存在"
- 铁律6拦截：空项目启动失败 → 阻断 → 提示"禁止编写业务代码"

### 维度二：对人（流程级约束）

**执行方式**：将8阶段做成Task模板，逐阶段打钩确认

**约束逻辑**：
1. 每阶段结束，弹出确认清单
2. 人工必须逐项确认
3. 未确认则无法进入下一阶段

**确认清单示例（阶段2）**：
- [ ] 我已审核业务流程图
- [ ] 我已确认目录结构
- [ ] 我已确认依赖清单
- [ ] 我已确认接口定义
- [ ] 我同意锁定以上内容

### 维度三：对流程（审计级追踪）

**执行方式**：阶段2锁档文件、阶段4启动截图提交Git审计

**追踪逻辑**：
1. 阶段2锁档文件提交Git
2. 阶段4启动截图提交Git
3. 每次代码提交附带阶段标记

**Git提交规范**：
- 阶段2：`[Stage-2] 锁档：流程图+目录树+依赖清单`
- 阶段4：`[Stage-4] 脚手架：空项目启动成功`
- 阶段5：`[Stage-5] 模块：{模块名称}开发完成`
- 阶段6：`[Stage-6] 测试：{模块名称}测试通过`

### 维度四：对质量（门禁级控制）

**执行方式**：阶段6异常测试覆盖率≥80%、阶段7 lint通过率写入CI门禁

**门禁逻辑**：
1. 阶段6：异常测试覆盖率≥80%
2. 阶段7：lint检查通过
3. 阶段7：全量测试通过

**门禁检查**：
- 覆盖率不达标 → 阻断 → 提示"请补充测试用例"
- lint不通过 → 阻断 → 提示"请修复代码风格问题"
- 测试失败 → 阻断 → 提示"请修复失败的测试"

## 铁律检查清单（每阶段执行）

### 阶段1-2
- [ ] 铁律3：未编造不存在的接口
- [ ] 铁律7：未编写死代码、占位代码

### 阶段3-4
- [ ] 铁律6：空项目初始化成功
- [ ] 铁律2：未修改阶段2锁档内容

### 阶段5
- [ ] 铁律1：单模块开发，未批量生成
- [ ] 铁律3：未编造接口
- [ ] 铁律4：已实现loading/empty/error状态
- [ ] 铁律5：未修改已验收模块
- [ ] 铁律7：未编写死代码

### 阶段6
- [ ] 铁律4：异常场景全覆盖
- [ ] 铁律5：修复未引入新Bug

### 阶段7
- [ ] 铁律7：已清理冗余代码
- [ ] 铁律5：重构未破坏已有功能

## 绝对禁令vs条件禁令

### 绝对禁令（零容忍，无例外）

- **铁律3**：编造不存在的接口——涉及运行时安全，无任何商量余地
- **铁律6**：空项目初始化失败就写业务代码——地基不牢，上面的一切都是废墟

### 条件禁令（有正式变更流程可豁免）

- **铁律2**：可通过"需求变更回溯"机制正式修改架构
- **铁律5**：可通过"需求变更单 + 重新验收"机制正式修改已验收模块

### 执行禁令（有场景适配弹性）

- **铁律1**：极简项目可简化为单模块
- **铁律4**：无UI的纯逻辑模块可豁免状态处理
- **铁律7**：第三方生成代码在阶段7统一清理

## 例外审批流程

```
发现需要例外
    │
    ▼
书面记录例外原因和影响范围
    │
    ▼
人工审批（不可委托给 AI）
    │
    ├──► 批准 → 执行变更 → 重新验证受影响模块 → 更新锁档文件
    │
    └──► 拒绝 → 按原铁律执行
```

关键原则：**例外必须是"正式流程"，不能是"口头默许"**。每一次例外都要留下审计痕迹。
