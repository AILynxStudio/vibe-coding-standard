# 阶段1：需求收敛

## 概述

阶段1是8阶段流程的起点，负责收集和整理项目需求，为后续开发奠定基础。

## 核心任务

1. 用<=100字描述项目用途、目标用户、核心痛点
2. 划分核心/次要/扩展功能
3. 判定项目体量与运行载体

## AI引导话术

```
让我们从需求开始。我会问您几个问题来帮助梳理需求。

基于Vibe Coding的最佳实践，我需要了解：

1. **项目目标**：这个项目要解决什么问题？
2. **目标用户**：谁会使用这个项目？
3. **核心功能**：最核心的功能是什么？（最多3个）
4. **次要功能**：有哪些次要功能？
5. **扩展功能**：有哪些扩展功能？（可选）
6. **项目体量**：小型/中型/大型？
7. **运行载体**：Web/移动端/桌面/CLI？

请回答这些问题，我会帮您整理成需求文档。
```

## 产出

- 极简项目简介（<=100字）
- 功能分级清单（核心/次要/扩展）
- 项目体量评估
- 运行载体确认

## 门禁检查

```
AI: 我已经整理了您的需求：
[需求文档内容]

请确认：
- [ ] 项目目标清晰
- [ ] 功能分级合理
- [ ] 项目体量评估准确
- [ ] 运行载体确认

请确认是否准确？如有需要修改的地方请告诉我。
```

## 用户可能的回复及AI应对策略

| 用户回复 | AI应对策略 |
|----------|-----------|
| **清晰描述需求** | 整理成需求文档，进入阶段2 |
| **模糊描述需求** | 追问具体问题："您说的XX具体指什么？" |
| **不确定需求** | 提供参考案例："您可以参考case-03的待办清单项目" |
| **需求过多** | 引导分级："请告诉我最核心的3个功能是什么？" |
| **需求过少** | 追问细节："除了XX功能，还有其他需求吗？" |

## 铁律检查

- 铁律3：未编造不存在的接口
- 铁律7：未编写死代码、占位代码

## 案例参考

### case-03：Cursor待办清单

**需求描述**：
```
开发一个个人待办清单（Todo List）Web应用。
核心功能：添加、完成、删除待办事项。
次要功能：筛选、编辑、本地存储。
项目体量：小型。
运行载体：Web。
```

**功能分级**：
```
核心功能：
1. 添加待办事项
2. 标记待办事项为已完成
3. 删除待办事项

次要功能：
1. 按状态筛选（全部/进行中/已完成）
2. 编辑待办事项
3. 数据本地存储（localStorage）

扩展功能：
1. 待办事项优先级（高/中/低）
2. 待办事项分类
3. 导出数据为JSON
```

## 常见问题

### Q: 需求不清晰怎么办？
A: 使用5W1H方法追问：
- What：要做什么？
- Why：为什么要做？
- Who：谁会使用？
- When：什么时候用？
- Where：在哪里用？
- How：怎么用？

### Q: 需求过多怎么办？
A: 引导用户分级：
1. 先确定核心功能（最多3个）
2. 再确定次要功能
3. 最后确定扩展功能

### Q: 需求过少怎么办？
A: 追问细节：
1. 除了核心功能，还有其他需求吗？
2. 有性能、安全、兼容性要求吗？
3. 有部署、运维、监控需求吗？
