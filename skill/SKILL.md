# Vibe Coding 智能引导式 SKILL

## 描述

这是一套深度融入Vibe Coding标准体系的智能引导式SKILL，通过AI主动提问、场景自动检测、8阶段流程引导，帮助用户完成项目开发。

## 核心特性

1. **四层解耦架构**：规则优先级清晰，上层约束下层
2. **7条铁律强制执行**：四维拦截机制，不只是检查而是强制
3. **工具包组合逻辑**：根据场景自动选择最佳组合
4. **案例实战智慧**：将最佳实践融入引导流程
5. **质量保障闭环**：每阶段有门禁，全程可追溯
6. **7种场景深度适配**：不同场景有不同的执行策略

## 使用方式

当用户请求使用此SKILL时，请按照以下流程执行：

### 1. 场景检测

首先询问用户：
```
你使用什么AI工具？
1. Cursor / Windsurf / Trae（智能IDE）
2. VSCode + AI插件（插件增强）
3. ChatGPT/Claude桌面版（桌面客户端）
4. Dify/Coze（低代码平台）
5. ChatGPT/Claude网页版（网页版AI）
6. AutoGen/AgentStack（多智能体）
```

### 2. 工具包组合推荐

根据用户选择的工具，推荐对应的工具包组合：

| 场景 | 推荐组合 | 部署方式 |
|------|----------|----------|
| 智能IDE | TK-1 + TK-2 + TK-5 | 文件部署 |
| 插件增强 | TK-1 + TK-2 + TK-5 | 文件部署+手动加载 |
| 桌面客户端 | TK-4 + TK-5 | 每次对话粘贴 |
| 低代码平台 | TK-6 | 平台固化 |
| 网页版AI | TK-3 | 系统提示词 |
| 多智能体 | TK-7 + TK-5 | 角色分发 |
| 企业级 | TK-1(组织级) + TK-5(CI化) | CI/CD集成 |

### 3. 8阶段流程引导

按以下8阶段流程引导用户完成项目开发：

#### 阶段1：需求收敛

AI主动提问：
```
让我们从需求开始。我会问您几个问题来帮助梳理需求：

1. **项目目标**：这个项目要解决什么问题？
2. **目标用户**：谁会使用这个项目？
3. **核心功能**：最核心的功能是什么？（最多3个）
4. **次要功能**：有哪些次要功能？
5. **扩展功能**：有哪些扩展功能？（可选）
6. **项目体量**：小型/中型/大型？
7. **运行载体**：Web/移动端/桌面/CLI？
```

产出：
- 极简项目简介（<=100字）
- 功能分级清单（核心/次要/扩展）
- 项目体量评估
- 运行载体确认

门禁：人工确认

#### 阶段2：架构锁档（核心锁档）

AI引导话术：
```
现在进入架构设计阶段。这是核心锁档阶段，一旦确认，后续阶段不得私自修改。

基于您的需求，我会：

1. 绘制业务流程图（Mermaid格式）
2. 设计模块拆分方案
3. 定义接口规范
4. 锁定目录结构
5. 整理依赖清单

请告诉我：
1. 您偏好的技术栈是什么？（如：React/Vue/Python等）
2. 有必须使用的第三方服务吗？
3. 有性能或安全方面的特殊要求吗？
```

产出：
- Mermaid业务流程图
- 架构说明文档
- 固定目录树
- 依赖清单

门禁：人工确认并锁定

#### 阶段3：环境对齐

AI引导话术：
```
现在检查开发环境是否就绪：

1. 检查您的AI工具是否已配置规则文件
2. 验证TK-1（项目硬约束）是否已加载
3. 确认TK-2（通用流程Skill）是否就位
4. 确认您理解增量开发模式

请告诉我您使用的AI工具，我会帮您配置。
```

检查清单：
- [ ] CLAUDE.md / .cursorrules 已创建
- [ ] TK-1 内容已写入规则文件
- [ ] TK-2 已部署到正确位置
- [ ] 增量开发模式已理解

门禁：人工确认

#### 阶段4：脚手架初始化（硬性门槛）

AI引导话术：
```
现在初始化项目脚手架：

1. 根据依赖清单初始化项目
2. 安装所有必要依赖
3. 清理冗余示例代码
4. 验证空项目能否成功启动

请运行以下命令：
[初始化命令]

然后告诉我运行结果。
```

硬性门槛：
```
⚠️ 硬性门槛：空项目必须成功启动且无报错，否则禁止进入编码阶段。

请确认：
- [ ] 项目初始化成功
- [ ] 依赖安装完成
- [ ] 冗余代码已清理
- [ ] 空项目启动无报错
```

门禁：空项目启动成功，人工确认

#### 阶段5：单模块增量编码

AI引导话术：
```
现在开始编码。我们将采用增量开发模式：

本次开发的模块：[模块名称]
模块功能：[功能描述]
预计代码量：[行数]

我会：
1. 只生成当前模块的代码
2. 生成后立即本地运行验证
3. 验证通过后再开发下一个模块

⚠️ 禁止：批量生成、多模块并行、全量代码生成

请确认开始开发这个模块？
```

编码节奏：
```
AI → 生成单模块代码
用户 → 运行验证
用户 → 确认通过
AI → 开发下一个模块
```

门禁：人工确认

#### 阶段6：单模块全场景测试

AI引导话术：
```
现在对模块进行全场景测试：

测试清单：
1. 正常流程测试
2. 空数据状态测试
3. 非法输入测试
4. 超时处理测试
5. 断连处理测试
6. Loading状态测试
7. Error状态测试
8. Empty状态测试

请逐项测试并告诉我结果。
```

门禁：测试覆盖率≥80%，人工确认

#### 阶段7：全局联调+重构优化

AI引导话术：
```
现在进行全局联调和重构：

1. 全业务链路联调
2. 统一代码风格
3. 消除冗余代码/死代码
4. 性能优化

请运行以下命令检查代码质量：
[lint命令]
[测试命令]

然后告诉我结果。
```

检查清单：
- [ ] 全业务链路联调通过
- [ ] 代码风格统一
- [ ] 冗余代码已清理
- [ ] 性能优化完成
- [ ] lint检查通过
- [ ] 全量测试通过

门禁：lint检查通过，全量测试通过，人工确认

#### 阶段8：打包交付+文档归档

AI引导话术：
```
项目即将完成！现在进行交付准备：

1. 按目标平台打包
2. 生成标准README
3. 制定迭代规划

请告诉我：
1. 目标平台是什么？（Web/移动端/桌面）
2. 需要部署到哪里？（GitHub Pages/Vercel/自有服务器）
3. 下一版本计划做什么功能？
```

交付清单：
- [ ] 项目已成功打包
- [ ] README已生成
- [ ] 迭代规划已制定
- [ ] 代码已提交到Git

门禁：人工确认

## 7条铁律（强制执行）

### 铁律1：禁止批量生成多模块
- 对抗问题：AI惰性 & 熵增失控
- 执行方式：检测到批量生成时阻断，提示"请单模块开发"

### 铁律2：阶段2定稿不得私自修改
- 对抗问题：架构随意变动
- 执行方式：检测到修改锁档文件时阻断，提示"锁档文件不可修改"

### 铁律3：严禁编造不存在的接口
- 对抗问题：AI幻觉
- 执行方式：检测到编造接口时阻断，提示"请先确认接口存在"

### 铁律4：必须补充完整状态处理
- 对抗问题：只写"快乐路径"
- 执行方式：检测到缺少状态处理时阻断，提示"请补充loading/empty/error状态"

### 铁律5：已验收模块禁止随意修改
- 对抗问题：修复引入新Bug
- 执行方式：检测到修改已验收模块时阻断，提示"已验收模块不可修改"

### 铁律6：空项目初始化失败禁止编写业务代码
- 对抗问题：在错误基础上建造
- 执行方式：空项目启动失败时绝对阻断

### 铁律7：禁止编写死代码、占位代码
- 对抗问题：代码腐烂
- 执行方式：检测到死代码时警告，提示"请清理死代码"

## 四层架构（优先级从高至低）

### L1：项目硬约束层（最高优先级）
- 载体：.cursorrules / CLAUDE.md
- 职责：管控目录结构、依赖、开发行为、Git规范、交付标准

### L2：通用流程Skill层
- 载体：vibe-coding-std
- 职责：统一8阶段流程、人机协作规则

### L3：技术规范层
- 载体：官网/GitHub原版Skill
- 职责：承载编码语法、框架规范

### L4：场景兜底层（最低优先级）
- 载体：TK-3/TK-4/TK-6
- 职责：无本地文件时的兜底方案

## 冲突解决机制

核心裁决规则：**上层约束下层，规则出现歧义时以更高层级为准。**

四步判断法：
1. 判断是否有"铁律"被违反？→ 是则立即停止执行
2. 判断当前执行处于哪个阶段？→ 检查阶段门禁
3. 判断规则冲突来自哪个层级？→ 上层优先
4. 无法自动裁决 → 报告给人类仲裁

## 常见陷阱及解决方案

### 陷阱1：AI批量生成多个文件
解决方案：立即停止，回滚到上一次提交，只保留目标模块的改动

### 陷阱2：AI编造不存在的接口
解决方案：要求AI先输出接口清单，人工核实后再编码

### 陷阱3：页面空白但没报错
解决方案：检查组件引入，把内容发给AI检查

### 陷阱4：阶段2锁档后需要修改
解决方案：必须回到阶段2重新定稿，并向下传播所有变更

## 最佳实践

1. **规则先行，代码后行**：先部署规则文件，再开始编码
2. **阶段确认，步步为营**：每阶段结束人工确认
3. **增量编码，单模块验证**：每次只开发一个模块
4. **测试清单，逐项验证**：使用测试清单逐项测试
5. **错误反馈，精确描述**：把报错发给AI，比描述问题更有效

## 注意事项

1. 8阶段流程必须串行执行，禁止跳步
2. 每阶段结束需要人工确认才能进入下一阶段
3. 铁律是强制执行的，违反时会阻断流程
4. 如需修改阶段2锁档内容，必须回到阶段2重新定稿
